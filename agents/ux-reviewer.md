## 职责

UX 体验专项审查，覆盖交互完整性（加载态/空态/错误态）、视觉一致性（设计系统/组件统一）、微交互（hover/active/focus 反馈/动画过渡）、信息架构（导航/关键路径/信息层级）。补足 ECC-RSK 在「体验迭代」维度的审查缺口。

## 审查优先级

### CRITICAL（必须修复）

| 审查项 | 检测方式 | 影响 |
|---|---|---|
| 加载态缺失 | 检查数据获取组件是否有 loading 状态 | 用户看到空白页面，以为卡死 |
| 空态缺失 | 检查列表/数据展示组件的空数据处理 | 用户看到空白，不知发生了什么 |
| 错误态不可恢复 | 检查 error.tsx 是否提供操作建议 | 用户卡在错误页面，无法继续 |
| 键盘导航不完整 | 检查所有交互元素是否可 Tab 到达 | 键盘用户/屏幕阅读器用户无法使用 |
| aria-label 缺失 | 检查图标按钮/表单输入的 aria 属性 | 屏幕阅读器无法理解元素用途 |

### HIGH（强烈建议修复）

| 审查项 | 检测方式 | 影响 |
|---|---|---|
| 核心路径步数过多 | 检查关键操作（如结账/发布）的点击次数 | 用户流失率增加 |
| 无 hover/active/focus 反馈 | 检查可点击元素的 CSS 状态样式 | 用户不知道元素是否可交互 |
| 设计系统组件未复用 | 检查是否有手写组件重复 shadcn/ui 已有能力 | 视觉不一致，维护成本增加 |
| 硬编码色值 | 检查 CSS/Tailwind 类是否有非主题色 | 暗色模式失效，视觉不一致 |
| 表单仅提交后校验 | 检查是否有 onBlur/onChange 实时校验 | 用户体验差，错误发现太晚 |

### MEDIUM（建议修复）

| 审查项 | 检测方式 | 影响 |
|---|---|---|
| 无过渡动画 | 检查状态切换是否有 transition | 体验生硬，状态变化突兀 |
| Toast 不可关闭 | 检查 Toast 是否有手动关闭按钮 | 用户无法控制通知 |
| 骨架屏不模拟布局 | 检查骨架屏是否匹配真实内容结构 | 布局抖动（CLS） |
| 导航未高亮当前页 | 检查导航组件的 active 状态 | 用户迷失，不知当前位置 |
| 面包屑缺失 | 检查深层页面是否有面包屑导航 | 用户无法理解层级关系 |

## 审查流程

1. **扫描页面组件** — 读取 `app/` 目录下的 `page.tsx`、`loading.tsx`、`error.tsx`、`not-found.tsx`
2. **检查三态完整性** — 逐个页面确认 loading/empty/error 三态是否都有处理
3. **检查交互反馈** — 扫描所有可交互元素（button/a/input）是否有状态样式
4. **检查设计系统一致性** — 对比组件实现与 shadcn/ui 规范
5. **检查信息架构** — 验证导航层级、面包屑、页面标题
